# 交付工作流

这份文档定义 SwellLobster 的默认开发与交付节奏。目标是让你自己和 AI 都按同一套方式推进工作，避免需求、实现、校验和沉淀脱节。

## 1. 开始任务前

- 先确认任务目标、边界和验收标准。
- 小任务直接在最近的 `AGENTS.md` 或相关文档里补充约束。
- 大任务先在 `docs/` 写一个短说明，至少包含：
  - 目标
  - 不做什么
  - 影响范围
  - 验收标准

如果任务会改变目录职责、数据模型或交互约束，先更新文档再动代码。

## 2. 实现阶段

- 只改和当前任务直接相关的目录。
- 优先遵循最近的 `AGENTS.md`，不要自己发明局部规范。
- 改动过程中如果发现“现有文档不再准确”，立即修正文档，不要留到最后。

默认执行顺序：

```bash
npm run verify:docs
npm run verify
```

局部试验可以先跑子目录命令，但准备提交前必须回到根命令闭环。

## 3. 收尾阶段

- 确认实现满足原始验收标准。
- 检查是否引入了新的约束、模式或坑点。
- 如果同类问题未来还会再出现，把它沉淀到：
  - 最近的 `AGENTS.md`
  - `docs/` 中的稳定文档
  - 测试
  - lint / consistency check / CI

原则：修过两次的问题，第三次不应再靠记忆避免。

## 4. 日常维护节奏

- 周期性回看 `AGENTS.md` 是否仍准确。
- 每次新增顶层目录或重要子系统时，同步补 `AGENTS.md`。
- 如果 README、`AGENTS.md`、代码三者开始分叉，以 `AGENTS.md` 和可执行脚本为准，并尽快修正文档漂移。

## 5. 默认协作规则

- README 负责对外说明项目。
- `AGENTS.md` 负责说明“如何在这个目录里安全地工作”。
- `CLAUDE.md` 仅保留兼容跳转，不再承载真实规则。
- 根 `verify` 是默认质量门；CI 是最终合并门。
